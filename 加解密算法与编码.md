# 加解密算法与编码

常见编码方式、哈希算法、对称/非对称加密的原理及其在安全测试中的应用。

---

## 一、编码方式

编码（Encoding）不是加密，只是将数据从一种格式转换为另一种格式，**不依赖密钥，可逆**。安全测试中常用于绕过过滤、隐藏载荷、数据传输。

### 1. Base64

每 3 字节编码为 4 个可打印 ASCII 字符，字符集 `A-Za-z0-9+/`，末尾用 `=` 填充。

```
原始:  hello
Hex:   68 65 6C 6C 6F
二进制拼接: 01101000 01100101 01101100 01101100 01101111
6位分组:    011010 000110 010101 101100 011011 000110 111100(00)
Base64:     a      G      V      s      b      G      8      =
结果: aGVsbG8=
```

| 特征 | 说明 |
|------|------|
| 字符集 | `A-Z` `a-z` `0-9` `+` `/` `=` |
| 长度 | 总是 4 的倍数 |
| 常见场景 | HTTP Basic Auth、Data URI、邮件 MIME、JWT、图片 Base64 内嵌 |
| 安全相关 | 编码恶意载荷绕过 WAF、PHP `base64_decode()` 后门 |

**Base64 变体：**

| 变体 | 差异 | 使用场景 |
|------|------|----------|
| Base64 URL | `+/` 替换为 `-_`，不填充 `=` | URL/JWT 中的 Base64 |
| Base32 | 字符集 `A-Z2-7`，末尾 `=` 填充 | 密钥恢复码、OTP 密钥 |
| Base16 (Hex) | 每字节 2 个十六进制字符 `0-9A-F` | 二进制数据的可读表示 |

### 2. URL 编码（百分号编码）

将特殊字符替换为 `%` 加两位十六进制值。

```
空格  → %20
<    → %3C
>    → %3E
'    → %27
"    → %22
/    → %2F
#    → %23
&    → %26
=    → %3D
```

| 安全相关 | 说明 |
|----------|------|
| 双重 URL 编码 | `%2527`（`%25` = `%`）绕过 WAF 对单引号的过滤 |
| Unicode URL 编码 | `%u0027` 部分后端可解析，用于绕过规则 |
| SQL 注入 | `%27` 替代单引号 |
| XSS | `%3Cscript%3E` 替换 `<script>` |

### 3. HTML 实体编码

在 HTML 中用特殊标记表示字符，防止被浏览器解析执行。

| 字符 | 命名实体 | 数值实体 |
|------|---------|----------|
| `<` | `&lt;` | `&#60;` |
| `>` | `&gt;` | `&#62;` |
| `&` | `&amp;` | `&#38;` |
| `"` | `&quot;` | `&#34;` |
| `'` | `&apos;` | `&#39;` |

| 安全相关 | 说明 |
|----------|------|
| XSS 绕过 | `&#x3c;script&#x3e;` 替代 `<script>` |
| 混淆编码 | 十进制 `&#60;` / 十六进制 `&#x3c;` 混合使用绕过黑名单 |

### 4. Unicode 编码

为全球所有字符分配唯一码点，有多种编码形式。

| 编码形式 | 示例（字符 `A`） | 特点 |
|----------|-----------------|------|
| UTF-8 | `41` | 变长编码，向后兼容 ASCII，Web 最常用 |
| UTF-16 LE | `41 00` | Windows 内部使用 |
| UTF-16 BE | `00 41` | Java 的 Modified UTF-8 |
| UTF-32 | `00 00 00 41` | 定长 4 字节 |

| 安全相关 | 说明 |
|----------|------|
| Unicode 同形字 | 用 Cyrillic `а`（U+0430）伪装 Latin `a`（U+0061）钓鱼 |
| 路径遍历绕过 | `..%c0%af` 等非法 UTF-8 序列绕过过滤 |
| 字符集绕过 | 不同字符集解析差异导致过滤绕过（如 GBK/UTF-8） |

### 5. 其他常见编码

| 编码 | 说明 | 示例 |
|------|------|------|
| **Hex（十六进制）** | 每字节两个十六进制字符 | `Hello` → `48656C6C6F` |
| **ASCII 码** | 0-127 号字符 | `A` = 65 |
| **Quoted-printable** | 邮件编码，`=` 加两位十六进制 | `=48=65=6C=6C=6F` |
| **MIME** | 邮件附件编码标准 | `=?UTF-8?B?` 开头 + Base64 |
| **Punycode** | 国际化域名编码 | `xn--fsq` → `中国` |
| **Shellcode hex** | `\x` 前缀十六进制 | `\x68\x65\x6C\x6C\x6F` |
| **JSON Unicode** | `\u` 前缀 | `hello` |
| **莫尔斯码** | 点和划表示字母 | SOS: `... --- ...` |

## 二、哈希算法

哈希（Hash）将任意长度数据映射为固定长度的摘要。**单向不可逆**，用于密码存储、数据完整性校验、数字签名。

### 1. 常见哈希算法对比

| 算法 | 输出长度 | 是否安全 | 当前状态 |
|------|----------|----------|----------|
| **MD5** | 128 bit（32 位 Hex） | 否 | 已被破解，存在碰撞攻击，仅用于非安全场景 |
| **SHA-1** | 160 bit（40 位 Hex） | 否 | 已被破解（SHAttered），不再推荐 |
| **SHA-256** | 256 bit（64 位 Hex） | 是 | 当前标准，应用广泛 |
| **SHA-512** | 512 bit（128 位 Hex） | 是 | 更高安全需求场景 |
| **SM3** | 256 bit | 是 | 中国国家密码标准 |

```
# 示例
MD5("hello")    = 5d41402abc4b2a76b9719d911017c592
SHA1("hello")   = aaf4c61ddcc5e8a2dabede0f3b482cd9aea9434d
SHA256("hello") = 2cf24dba5fb0a30e26e83b2ac5b9e29e1b161e5c1fa7425e73043362938b9824
```

### 2. 哈希碰撞

两个不同输入产生相同的哈希值，说明该哈希算法在密码学上已破解。

| 年份 | 算法 | 事件 |
|------|------|------|
| 2004 | MD5 | 王小云团队成功构造碰撞 |
| 2017 | SHA-1 | Google 公布首个 SHA-1 碰撞（SHAttered） |
| 至今 | SHA-256 | 未发现有效碰撞 |

### 3. 哈希在安全领域的应用

| 场景 | 说明 |
|------|------|
| **密码存储** | 数据库不存明文，存 `hash(password + salt)` |
| **文件完整性** | 发布软件附 SHA256 校验值，防止篡改 |
| **数字签名** | 对原数据哈希后再用私钥签名（性能优化） |
| **区块链** | 交易哈希、区块哈希、Merkle Tree |
| **API 签名** | 请求参数拼接待签名字符串 + 哈希防篡改 |

### 4. 加盐哈希（Salt & Pepper）

```
存储: bcrypt(password + salt) = $2a$12$...

不加盐的问题:
  相同密码 → 相同哈希 → 彩虹表一查全出来

加盐后:
  admin + "abc123"  → hash1
  admin + "xyz789"  → hash2  (不同盐值，哈希完全不同)
```

| 概念 | 含义 | 存储位置 |
|------|------|----------|
| **Salt（盐）** | 每个用户唯一的随机值 | 和哈希一起存在数据库 |
| **Pepper（胡椒）** | 全局统一的密钥 | 存在代码中或密钥管理服务，不进数据库 |

**彩虹表攻击：** 预先计算常见密码及对应哈希的巨型查询表。加盐使相同密码产生不同哈希，彩虹表直接失效。

### 5. 密码专用哈希

| 算法 | 特点 |
|------|------|
| **Bcrypt** | 内置 Salt，可调节迭代次数（cost factor），抗暴力破解 |
| **Scrypt** | 同时消耗 CPU 和内存，抗硬件加速 |
| **Argon2** | 2015 年密码哈希竞赛冠军，抗 GPU/ASIC/FPGA |
| **PBKDF2** | NIST 推荐，多次迭代 HMAC |

### 6. HMAC（键控哈希消息认证码）

```
HMAC(key, message) = Hash((key ⊕ opad) || Hash((key ⊕ ipad) || message))
```

| 验证码变量 | 说明 |
|-----------|------|
| **MAC** | 纯密钥参与哈希 |
| **HMAC** | 密钥参与哈希，结构更安全（防长度扩展攻击） |
| **NMAC** | 嵌套 MAC |
| **CMAC** | 使用 CBC-MAC 变体 |

| 应用 | 说明 |
|------|------|
| JWT 签名 | `HMAC-SHA256(secret, base64(header) + "." + base64(payload))` |
| API 签名 | 对请求体做 HMAC 防篡改 |
| Cookie 完整性 | 签 Cookie 防伪造 |

## 三、对称加密

**加密和解密使用相同的密钥。** 速度快，适合大数据量加密，难点在于密钥如何安全分发。

### 1. 加密模式

| 模式 | 全称 | 特点 | 安全性 |
|------|------|------|--------|
| **ECB** | Electronic Codebook | 相同明文 → 相同密文（每个分组独立加密） | 不安全 |
| **CBC** | Cipher Block Chaining | 前一个密文块与当前明文块 XOR 后再加密 | 需要 IV |
| **CTR** | Counter | 将分组加密变成流加密，计数器递增加密后 XOR 明文 | 可并行 |
| **CFB** | Cipher Feedback | 类似 CBC，流式处理 | 需要 IV |
| **OFB** | Output Feedback | 密钥流独立于明文，加密解密相同操作 | 不需要填充 |
| **GCM** | Galois/Counter Mode | CTR + 认证标签，同时提供加密和完整性校验 | 推荐 |

```
ECB 的安全缺陷: 相同 16 字节明文块产生相同密文块

原始图片（BMP）使用 AES-ECB 加密后，
像素分布模式仍然可见！
→ 必须避免使用 ECB 模式
```

### 2. 主流对称加密算法对比

| 算法 | 密钥长度 | 块大小 | 状态 |
|------|----------|--------|------|
| **DES** | 56 bit | 64 bit | 已淘汰，可暴力破解 |
| **3DES** | 112/168 bit | 64 bit | 不推荐，慢且有安全问题 |
| **AES** | 128/192/256 bit | 128 bit | 当前标准 |
| **SM4** | 128 bit | 128 bit | 中国国家密码标准 |
| **RC4** | 40-2048 bit | 流加密 | 已淘汰，已有攻击 |
| **ChaCha20** | 256 bit | 流加密 | 移动端/低功耗设备首选 |
| **Blowfish** | 32-448 bit | 64 bit | 已淘汰 |
| **Twofish** | 128/192/256 bit | 128 bit | Blowfish 继任者 |

**AES 在 Web 安全中的常见场景：**
- HTTPS/TLS 中的数据加密
- Cookie 中的加密字段（如 `remember_me` 系列反序列化漏洞）
- 数据库敏感字段加密
- 加密配置文件中的密码

### 3. 填充（Padding）

块加密要求明文长度是块大小的整数倍，不足时需要填充。

| 填充方式 | 说明 |
|----------|------|
| **PKCS#7** | 缺 N 字节就填 N 个值为 N 的字节（最常用） |
| **PKCS#5** | PKCS#7 的特例，仅用于 8 字节块 |
| **Zero Padding** | 缺多少个字节填多少个 `00` |
| **ISO 10126** | 随机字节 + 最后 1 字节表示填充长度 |
| **ANSI X.923** | `00` 填充 + 最后 1 字节表示填充长度 |

**Padding Oracle 攻击：**
某些实现根据解密后 Padding 是否正确返回不同的错误信息，攻击者可逐字节推断出明文内容。CVE-2013-0169（Lucky13）、经典的 ASP.NET Padding Oracle 漏洞都属此类。

> 使用 GCM/CCM 等 AEAD 模式可避免 Padding Oracle。

## 四、非对称加密

**加密和解密使用不同的密钥**：公钥加密、私钥解密。解决对称加密的密钥分发问题。

```
发送方（Alice）用 Bob 的公钥加密 → 密文 → Bob 用自己的私钥解密

- 公钥可以公开，谁都能用
- 私钥必须保密，只有持有者能解密
```

### 1. 主流非对称加密算法对比

| 算法 | 密钥长度 | 安全性基础 | 特点 |
|------|----------|-----------|------|
| **RSA** | 2048/4096 bit | 大整数因数分解 | 最广泛使用，速度较慢 |
| **ECC** | 256 bit | 椭圆曲线离散对数 | 短密钥实现高安全性，主流趋势 |
| **SM2** | 256 bit | 椭圆曲线 | 中国国家密码标准 |
| **ElGamal** | 1024-4096 bit | 离散对数 | 基于 Diffie-Hellman |
| **Ed25519** | 256 bit | 椭圆曲线（Edwards 曲线） | 现代 SSH/GPG 推荐 |

**RSA vs ECC 安全性对比：**

| 安全强度 | RSA 密钥长度 | ECC 密钥长度 |
|----------|-------------|-------------|
| 128 bit | 3072 bit | 256 bit |
| 256 bit | 15360 bit | 512 bit |

### 2. 非对称加密的实际应用

| 场景 | 说明 |
|------|------|
| **HTTPS/TLS** | 非对称交换对称密钥 + 对称加密传输数据 |
| **SSH** | RSA/Ed25519 密钥登录，公钥放服务器，私钥本地 |
| **JWT** | RS256 签名：私钥签发、公钥验签 |
| **GPG/PGP** | 邮件加密、文件签名 |
| **数字货币** | 公钥 = 地址，私钥 = 掌控资产 |
| **数字证书** | CA 证书链验证网站身份 |

### 3. 密钥交换协议 —— Diffie-Hellman

在不安全的信道上协商出共享密钥，即使中间人窃听也无法推导。

```
Alice                    公共信道                    Bob
选择私钥 a                公开 p, g                 选择私钥 b
计算 A = g^a mod p  ───  A  ───→
                 ←─── B  ───                     计算 B = g^b mod p
共享密钥 = B^a mod p                             共享密钥 = A^b mod p

最终: g^(ab) mod p  =  g^(ba) mod p  (双方密钥相同!)
窃听者只知道 g, p, A, B，无法高效计算出 g^(ab)
```

## 五、数字签名

**用私钥签名、用公钥验签**，实现：身份认证 + 数据完整性 + 不可否认性。

```
签名:  hash(原文) → 用私钥加密哈希 → 数字签名
验签:  用公钥解密签名 → 得到 hash1 → 重新 hash(原文) → 得到 hash2 → 对比
```

| 签名算法 | 哈希搭配 | 使用场景 |
|----------|----------|----------|
| RSA-PSS | SHA-256/512 | JWT RS256/RS512 |
| ECDSA | SHA-256 | JWT ES256、比特币交易签名 |
| EdDSA | SHA-512 | SSH Ed25519 |
| SM2 + SM3 | SM3 | 中国商密体系 |

**JWT 签名算法族：**

| JWT 算法 | 说明 |
|----------|------|
| `HS256` | HMAC-SHA256，对称，secret 泄露即被伪造 |
| `RS256` | RSA-SHA256，非对称，私钥签发公钥验签 |
| `ES256` | ECDSA + SHA256 |
| `none` | 无签名，常见攻击：修改 header 的 `alg: none` |

## 六、编码/加密总结

```
数据安全处理流程:

编码 (Encoding)
├── 无密钥，纯粹格式转换
├── Base64 / Hex / URL Encode / HTML Entity
└── 用途: 数据传输、字符集转换、载荷混淆

       ↓ (加上密钥)

加密 (Encryption)
├── 对称加密: 同一个密钥加解密
│   ├── AES（CBC / GCM）— 当前标准
│   └── 优点快，缺点密钥分发困难
│
├── 非对称加密: 公钥加密、私钥解密
│   ├── RSA / ECC — 解决密钥分发
│   └── 优点安全，缺点速度慢
│
└── 混合: HTTPS = 非对称交换对称密钥 + 对称加密传输数据

       ↓ (单向不可逆)

哈希 (Hash)
├── MD5 / SHA-1 — 已淘汰
├── SHA-256 / SHA-512 — 当前标准
├── Bcrypt / Argon2 — 密码存储专用
└── HMAC — 带密钥的消息认证
```

## 七、安全测试中需要注意的加密问题

| 问题 | 说明 |
|------|------|
| **弱算法识别** | 发现 Web 应用使用 MD5 存密码、AES-ECB 加密 Cookie，可直接报告高危 |
| **硬编码密钥** | 前端 JS 中搜索 `AES_KEY`、`SECRET`、`privateKey` 等常量 |
| **ECB 模式检测** | 相同输入产生相同输出 → Cookie 注入攻击、重放攻击 |
| **弱密钥长度** | 1024 位 RSA、DES 56 位密钥已可被破解 |
| **Padding Oracle** | 错误提示区分 PaddingError 和 DecryptError → 可解密任意密文 |
| **JWT `alg=none`** | 修改 JWT Header 为 `{"alg":"none"}` 可伪造任意 JWT |
| **密钥泄露** | `.git/config`、`.env` 泄露、Android APK 硬编码密钥 |
| **伪随机数** | 使用 `rand()` 而非密码学安全的随机数生成器（CSPRNG） |

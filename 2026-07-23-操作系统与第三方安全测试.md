# 2026-07-23 操作系统、数据库与第三方安全测试

安全测试不仅仅是 Web 层面，服务器上运行的操作系统、数据库、第三方软件同样可能成为突破口。
今天梳理了这三个层面的识别方法、漏洞类型和测试思路。

---

## 一、操作系统层面

### 1. 识别操作系统的常见方法

| 方法 | 具体操作 | 准确度 |
|------|----------|--------|
| **TTL 值判断** | `ping <目标IP>`，观察 TTL 返回值 | 粗略 |
| **Nmap -O** | `nmap -O <目标IP>`，TCP/IP 栈指纹匹配 | 高 |
| **抓包分析** | 分析 TCP 窗口大小、初始 TTL、IP ID 序列等 | 高 |
| **Banner 信息** | 连接 SSH、FTP 等服务，看返回的欢迎信息 | 中 |
| **HTTP 响应头** | 看 `Server` 字段暴露的信息 | 中 |
| **错误页面** | 访问不存在页面，触发 404/500 错误可能泄露 Server 信息 | 低 |

**TTL 值快速判断：**

| TTL 初始值 | 常见操作系统 |
|-----------|-------------|
| 64 左右 | Linux / Unix / macOS / Android |
| 128 左右 | Windows |
| 255 左右 | 网络设备（Cisco、路由器、防火墙） |

### 2. Windows 与 Linux 的核心区别及识别意义

| 维度 | Windows | Linux |
|------|---------|-------|
| **文件系统** | NTFS / FAT32 | ext4 / XFS / Btrfs |
| **路径分隔符** | `\`（反斜杠） | `/`（正斜杠） |
| **大小写敏感** | 不敏感（a = A） | 敏感（a ≠ A） |
| **常见 Web 服务** | IIS | Nginx / Apache |
| **常见数据库** | MSSQL | MySQL / PostgreSQL |
| **远程管理** | RDP（3389） | SSH（22） |
| **用户管理** | 域 / 本地用户 | UID/GID / root |
| **命令行** | cmd / PowerShell | bash / sh / zsh |
| **权限控制** | ACL / UAC | chmod / sudo |

**为什么要区分操作系统？**
- **漏洞方向不同**：Windows 关注 RDP、SMB、IIS 漏洞；Linux 关注 SSH、内核提权、Nginx/Apache 漏洞
- **利用方式不同**：反弹 Shell 的命令不同（`cmd.exe` vs `/bin/bash`）
- **提权路径不同**：Windows 走内核漏洞、服务权限、令牌窃取；Linux 走 SUID、sudo 配置、定时任务
- **后续横向移动方式不同**：Windows 域环境可用 Pass-the-Hash；Linux 通过 SSH 密钥传播

### 3. 操作系统层面常见漏洞类型

| 漏洞类型 | 典型代表 | 风险等级 |
|----------|----------|----------|
| **RCE 漏洞** | MS17-010（永恒之蓝）、CVE-2019-0708（蓝洞）、OpenSSL 心脏滴血 | 严重 |
| **权限提升** | Dirty Pipe（Linux）、PrintSpoofer（Windows）、内核提权 | 高 |
| **弱口令爆破** | RDP 弱口令、SSH 弱口令、FTP 弱口令 | 高 |
| **未授权访问** | SMB 共享未授权、Redis 无密码开放公网、各类服务默认口令 | 高 |
| **信息泄露** | NetBIOS 泄露主机名/用户名、SNMP 团体字泄露 | 中 |
| **补丁缺失** | 未打补丁且已公开利用方式的漏洞（Nday） | 严重 |
| **配置缺陷** | 不必要的服务端口暴露、危险端口直接对公网开放 | 中 |

**重点关注的 Windows 高危端口：**

| 端口 | 服务 | 攻击方向 |
|------|------|----------|
| 135 / 139 / 445 | RPC / NetBIOS / SMB | 远程代码执行（永恒之蓝）、密码爆破、共享枚举 |
| 3389 | RDP | 弱口令爆破、蓝洞 RCE、中间人攻击 |
| 1433 | MSSQL | 弱口令、XP_CMDSHELL 命令执行 |

**重点关注的 Linux 高危端口/服务：**

| 端口 | 服务 | 攻击方向 |
|------|------|----------|
| 22 | SSH | 弱口令爆破、密钥泄露 |
| 3306 | MySQL | 弱口令、SQL 注入综合利用 |
| 6379 | Redis | 未授权访问 → 写 SSH 公钥、写 Webshell、写定时任务 |
| 2375 | Docker API | 未授权访问 → 容器逃逸 |

### 4. 操作系统层面漏洞的影响范围

- **获取服务器完全控制权**：拿到最高权限（root / SYSTEM）
- **横向移动到内网**：以被控机器为跳板，攻击同网段的其他服务器
- **作为持久化据点**：植入后门、创建隐蔽账号、设置定时任务，长期保持访问
- **数据窃取与破坏**：拖库、加密勒索、删库跑路

> 操作系统一旦失陷，上面的所有应用（Web、数据库、缓存）全部沦陷。这是安全测试中收益最大、也是危害最高的环节。

## 二、数据库层面

### 1. 识别数据库类型的常见方法

| 方法 | 说明 |
|------|------|
| **端口扫描** | 每种数据库有默认监听端口，通过端口推断数据库类型 |
| **SQL 注入报错** | 不同数据库的报错信息有独特特征，可通过报错判断类型 |
| **Banner 信息** | 直接连接数据库服务获取欢迎信息 |
| **SQL 语法差异** | 利用不同数据库的独有函数/语法进行指纹识别 |
| **Web 应用特征** | CMS 类型可反推数据库（WordPress → MySQL） |

**常用数据库默认端口：**

| 数据库 | 默认端口 |
|--------|---------|
| MySQL | 3306 |
| MSSQL（SQL Server）| 1433 |
| Oracle | 1521 |
| PostgreSQL | 5432 |
| Redis | 6379 |
| MongoDB | 27017 / 27018 |
| Elasticsearch | 9200 |
| Memcached | 11211 |

**SQL 注入时通过函数指纹判断数据库类型：**

| 数据库 | 字符串拼接 | 注释符 | 特有函数/语法 |
|--------|-----------|--------|-------------|
| MySQL | `CONCAT()` 或 `||` | `#` `--` `/**/` | `sleep()` `load_file()` `into outfile` |
| MSSQL | `+` | `--` `/**/` | `xp_cmdshell` `waitfor delay` |
| Oracle | `\|\|` | `--` `/**/` | `utl_http.request()` `dbms_pipe.receive_message()` |
| PostgreSQL | `\|\|` | `--` `/**/` | `pg_sleep()` `COPY ... FROM PROGRAM` |
| Access | `&` | 无 | `CDate()` `IIF()` |

### 2. 数据库类型区别及识别意义

| 数据库 | 特点 | 常见搭配场景 |
|--------|------|-------------|
| **MySQL** | 开源、最流行的 Web 后端数据库 | PHP + Linux 站（LAMP/LNMP）|
| **MSSQL** | 微软出品，与 Windows/AD 深度集成 | ASP/ASPX/.NET + Windows 站 |
| **Oracle** | 大型企业级，功能强大但复杂 | Java 大型企业应用、金融系统 |
| **PostgreSQL** | 开源高级数据库，功能接近 Oracle | Python / Ruby / Go 应用 |
| **Redis** | 内存键值存储，常用于缓存/会话 | 各种 Web 应用的缓存层 |
| **MongoDB** | 文档型 NoSQL，JSON 格式存储 | Node.js / 大数据场景 |
| **Access** | 微软桌面数据库（.mdb 文件） | 小型 ASP 站点 |

**识别意义：**
- 不同数据库的**攻击方式**完全不同（注入语法、存储过程、利用工具都不同）
- 不同数据库的**提权路径**不同（MSSQL 可通过 xp_cmdshell 直接执行系统命令；MySQL 通常需要结合文件写入漏洞）
- 数据库类型暗示了**后端语言和操作系统**，帮助缩小攻击面

### 3. 数据库常见漏洞类型及攻击

| 漏洞类型 | 说明 | 典型案例 |
|----------|------|----------|
| **SQL 注入** | 未过滤用户输入，恶意拼接 SQL 语句 | 登录绕过、数据脱库、写入 Webshell |
| **弱口令/默认口令** | 数据库使用弱密码或出厂默认密码 | root/root、sa/sa、system/manager |
| **未授权访问** | 数据库对外开放且无需认证 | Redis 默认无密码、MongoDB 旧版默认无认证 |
| **CVE 漏洞** | 数据库软件本身的代码执行漏洞 | MySQL 任意文件读取、PostgreSQL 命令执行 |
| **存储过程滥用** | 利用数据库自带的危险存储过程 | MSSQL `xp_cmdshell` 执行系统命令 |
| **文件读写** | 利用数据库的文件操作函数 | MySQL `load_file()` 读文件、`INTO OUTFILE` 写 Webshell |
| **NoSQL 注入** | MongoDB/Redis 等非关系型数据库的注入 | `$ne` `$gt` 绕过、JSON 注入 |

**Redis 未授权访问三种利用方式：**

```
1. 写 SSH 公钥 → SSH 登录服务器
2. 写 Webshell → 通过 Web 访问执行命令
3. 写定时任务（crontab）→ 反弹 Shell
```

**MSSQL xp_cmdshell 执行命令：**
```sql
-- 启用 xp_cmdshell（如果被禁用）
EXEC sp_configure 'show advanced options', 1;
RECONFIGURE;
EXEC sp_configure 'xp_cmdshell', 1;
RECONFIGURE;

-- 执行系统命令
EXEC xp_cmdshell 'whoami';
```

### 4. 数据库层面漏洞的影响范围

- **数据泄露**：用户信息、密码哈希、交易记录被窃取
- **数据篡改/删除**：导致业务异常、数据丢失
- **权限提升**：从数据库权限提升到操作系统权限（通过存储过程或文件操作）
- **横向移动**：数据库服务器往往是内网核心节点，拿下后可进一步探测内网

> 数据库漏洞的"连锁反应"很强，一个 SQL 注入最终可能通向服务器完全控制。

## 三、第三方平台/软件层面

### 1. 如何判断目标使用了哪些第三方平台或软件

| 方法 | 说明 |
|------|------|
| **端口扫描** | 通过开放端口判断运行的服务（如 2181 → Zookeeper，9200 → Elasticsearch） |
| **HTTP 响应头/Cookie** | 如 `X-Powered-By: PHP/7.4`、`Set-Cookie: laravel_session=` |
| **页面特征/版权信息** | 页面底部"Powered by XXX"、favicon 哈希匹配 |
| **特定路径访问** | `/phpmyadmin/`、`/swagger-ui.html`、`/actuator/health` |
| **源码/静态文件分析** | JS 中引用的组件路径、CSS 注释中的框架信息 |
| **Wappalyzer / WhatWeb** | 使用浏览器插件或命令行工具自动识别指纹 |
| **错误信息泄露** | 应用报错可能暴露框架、组件、版本信息 |
| **目录扫描** | 扫描后台路径、管理页面、API 文档等 |

### 2. 为什么要识别第三方平台或软件

- **已知漏洞直接利用**：识别出 Struts2 → 直接找 S2 系列 RCE；识别出 ThinkPHP 5.x → 找历史 RCE
- **缩小测试范围**：不是每个站点都需要从头测 SQL 注入，如果是公开 CMS，直接查已知漏洞更高效
- **判断攻击深度**：知道用的是 Redis，可以关注未授权；知道用的是 Fastjson，可以关注反序列化
- **串联信息**：不同第三方软件的信息可以拼凑出完整的服务器架构图

### 3. 常见第三方平台/软件漏洞类型及攻击

| 软件类型 | 典型代表 | 常见漏洞 |
|----------|----------|----------|
| **Web 中间件** | Apache、Nginx、IIS、Tomcat、WebLogic | 解析漏洞、目录穿越、反序列化、弱口令 |
| **CMS** | WordPress、DedeCMS、帝国 CMS | SQL 注入、后台弱口令、任意文件上传、插件漏洞 |
| **开发框架** | Struts2、ThinkPHP、Spring、Fastjson | RCE、反序列化、表达式注入 |
| **数据库管理工具** | phpMyAdmin、Adminer | 弱口令、未授权访问、代码执行 |
| **缓存/消息队列** | Redis、RabbitMQ、ActiveMQ | 未授权访问、反序列化、弱口令 |
| **搜索服务** | Elasticsearch、Solr | 未授权访问、RCE、信息泄露 |
| **大数据组件** | Hadoop、Kafka、Zookeeper | 未授权访问、RCE |
| **CI/CD 工具** | Jenkins、GitLab | 未授权访问、RCE、凭证泄露 |
| **VPN/远程访问** | OpenVPN、AnyConnect、Fortinet VPN | CVE 漏洞、弱口令 |
| **虚拟化管理** | vSphere ESXi、Docker、K8s API | 未授权访问、容器逃逸、CVE 漏洞 |
| **邮件服务** | Exchange、Zimbra、Coremail | SSRF、RCE、信息泄露 |
| **负载均衡** | F5 BIG-IP、Nginx、HAProxy | CVE 漏洞、配置缺陷 |
| **OA/办公系统** | 泛微、致远、通达 OA | SQL 注入、RCE、任意文件上传 |

### 4. 第三方平台/软件安全测试的范围

不是只有 Web 应用才需要渗透测试，服务器上运行的所有服务都是潜在入口：

- **Web 相关**：Web 服务器、WAF、CDN、反向代理
- **数据库相关**：数据库本体、缓存、搜索引擎、管理工具
- **远程管理**：SSH、RDP、VNC、TeamViewer、堡垒机
- **邮件服务**：SMTP、POP3、Exchange、Webmail
- **文件服务**：FTP、SMB、NFS、Rsync
- **中间件**：消息队列、ESB、定时任务调度
- **监控运维**：Zabbix、Nagios、Grafana、Prometheus
- **容器编排**：Docker API、Kubernetes API
- **网络设备**：路由器、交换机、防火墙、负载均衡

**测试原则：** 每个暴露的服务都可能存在漏洞（弱口令、未授权访问、已知 CVE、配置缺陷），不放过任何一个开放端口。

## 四、补充：安全测试的全局视角

### 测试不仅仅是 Web

常规 Web 安全测试（SQL 注入、XSS、文件上传等）只是安全测试的一个子集。一个完整的渗透测试应该覆盖：

```
                    目标系统
                       │
        ┌──────────────┼──────────────┐
        │              │              │
    操作系统层       Web 应用层      第三方服务层
   （OS + 端口）    （源码 + CMS）   （中间件 + 组件）
        │              │              │
   · 永恒之蓝      · SQL 注入      · 反序列化
   · SSH 爆破      · XSS           · 未授权访问
   · 内核提权      · 文件上传      · CVE 漏洞
   · 弱口令        · 逻辑漏洞      · 弱口令
```

### 非典型 Web 服务的安全测试

有些目标不提供 Web 服务，仅提供单一或特定的网络服务，这些同样可以且应该被测试：

| 服务类型 | 示例 | 测试方向 |
|----------|------|----------|
| 邮件服务 | SMTP（25）、POP3（110）、IMAP（143） | 弱口令爆破、邮件伪造、SMTP 注入、Exchange CVE |
| 游戏服务 | 游戏服务器端口 | 协议逆向、封包篡改、逻辑漏洞 |
| 负载均衡 | F5 BIG-IP、Nginx | CVE 漏洞（如 F5 BIG-IP RCE）、配置缺陷 |
| 文件传输 | FTP（21）、Rsync（873） | 匿名登录、弱口令、目录遍历 |
| 数据库 | MySQL（3306）、MSSQL（1433） | 弱口令、未授权访问、存储过程利用 |
| 远程管理 | SSH（22）、RDP（3389）、VNC（5900） | 弱口令爆破、CVE 漏洞 |
| DNS | DNS（53） | DNS 域传送漏洞、DNS 隧道 |
| 打印服务 | IPP（631）、RAW（9100） | 打印机弱口令、打印任务窃取 |

**核心思路不变：** 不管目标是 Web 应用、API 接口、还是单纯的服务器服务，安全测试的底层逻辑都是一致的——**信息收集 → 版本识别 → 漏洞匹配 → 漏洞利用 → 权限维持**。只是具体的技术栈和工具不同而已。

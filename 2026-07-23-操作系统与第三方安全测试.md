# 2026-07-23 操作系统、数据库与第三方安全测试

安全测试不仅仅是 Web 层面，服务器上运行的操作系统、数据库、第三方软件同样可能成为突破口。
今天梳理了这三个层面的识别方法、漏洞类型和测试思路。

---

## 前置：渗透测试核心流程

在深入各个层面之前，先明确渗透测试的整体思路。攻击一个服务器不是上来就乱打，而是有清晰的阶段划分：

```
┌──────────────────────────────────────────────────────────────────────┐
│                        渗透测试五阶段                                  │
├─────────────┬─────────────┬─────────────┬─────────────┬───────────────┤
│  1. 信息收集  │  2. 漏洞发现  │  3. 漏洞利用  │  4. 权限提升  │  5. 横向移动 /  │
│  Recon       │  Discovery  │ Exploitation │  Privilege  │   权限维持      │
│             │             │             │  Escalation │  Lateral Move  │
├─────────────┼─────────────┼─────────────┼─────────────┼───────────────┤
│ 端口扫描     │ 查 CVE 编号  │ 使用 PoC/Exp │ 内核提权     │ 密码抓取       │
│ OS 识别     │ Vulhub 复现  │ SQL 注入    │ SUID 提权   │ SSH 密钥窃取   │
│ 服务版本探测  │ Exploit-DB  │ 文件上传    │ UAC 绕过    │ 域渗透         │
│ 目录扫描     │ 手工测试     │ 反序列化 RCE │ 令牌窃取     │ ARP 欺骗       │
│ CMS 指纹    │ 代码审计     │ 弱口令登录   │ 计划任务     │ 内网扫描       │
└─────────────┴─────────────┴─────────────┴─────────────┴───────────────┘
                                    │
                                    │ 每推进一层，都可能发现新目标
                                    ▼ 回到阶段 1，对新目标重新走流程
```

### 形象类比

把渗透测试理解成"入室盗窃"的过程：

| 阶段 | 专业术语 | 通俗理解 |
|------|----------|----------|
| 1. 信息收集 | Recon | 在房子外面观察——有几扇门？什么锁？有没有报警器？有没有保镖？ |
| 2. 漏洞发现 | Discovery | 发现二楼窗户没关紧、地下室锁生锈了、报警器型号有公开绕过方法 |
| 3. 漏洞利用 | Exploitation | 从没锁的窗户爬进去，或者用工具撬开后门 |
| 4. 权限提升 | Privilege Escalation | 进去后发现只在一楼厨房，但主人房在二楼——想办法上楼 |
| 5. 横向移动 | Lateral Movement | 从主人房找到钥匙，打开隔壁单元的门，继续偷下一家 |

### 各阶段详细说明

#### 阶段一：信息收集（最重要的一步）

信息收集的深度直接决定了后续所有阶段的成功率。分为两部分：

**被动信息收集（不直接接触目标）：**

| 方法 | 收集内容 |
|------|----------|
| 搜索引擎（Google Hacking） | 子域名、敏感文件、后台地址 |
| Shodan / Fofa / ZoomEye | 开放端口、服务 Banner、SSL 证书 |
| Whois 查询 | 注册者信息、DNS 服务器、IP 范围 |
| 社工库查询 | 历史泄露的账号密码 |
| GitHub 搜索 | 源码泄露、配置文件、密码硬编码 |

**主动信息收集（直接与目标交互）：**

| 方法 | 工具 | 收集内容 |
|------|------|----------|
| 端口扫描 | Nmap / Masscan | 开放了哪些端口和服务 |
| OS 指纹识别 | Nmap -O | 操作系统类型和版本 |
| 服务版本探测 | Nmap -sV | 各服务的具体版本号 |
| 目录/文件扫描 | Dirsearch / gobuster / ffuf | 隐藏目录、后台路径、备份文件 |
| CMS 指纹识别 | WhatWeb / Wappalyzer | CMS 类型、框架、第三方组件 |
| WAF 检测 | wafw00f | 是否存在 Web 应用防火墙 |

> **关键认知：** 信息收集是你和目标之间的信息差。你知道的越多，你打的越准。而目标对你一无所知。

#### 阶段二：漏洞发现

拿到信息后，逐一排查可利用的突破口：

| 信息来源 | 如何找漏洞 |
|----------|-----------|
| **端口/服务版本** | 查该版本的 CVE 编号 → Exploit-DB / Vulhub 找利用脚本 |
| **CMS/框架版本** | 查该 CMS 的历史漏洞列表（DedeCMS 各版本漏洞、Struts2 S2 系列等） |
| **Web 应用** | 手工测试 SQL 注入、XSS、文件上传、逻辑漏洞 |
| **弱口令** | 针对 SSH、RDP、FTP、数据库、后台管理页进行口令爆破 |
| **未授权访问** | 测试 Redis、MongoDB、Docker API 等默认无认证的服务 |

#### 阶段三：漏洞利用

| 漏洞类型 | 利用收益 |
|----------|----------|
| RCE（远程代码执行） | 直接获得服务器控制权 |
| 文件上传 | 上传 Webshell，获得 Web 权限 |
| SQL 注入 | 拖库、读文件、写 Webshell |
| 弱口令 | 登录服务器 / 数据库 / 后台 |
| SSRF | 探测内网、读取云元数据 |

#### 阶段四：权限提升

Web 权限 ≠ 最高权限。拿到低权限后要想办法提权：

| 操作系统 | 常见提权路线 |
|----------|-------------|
| **Linux** | SUID 提权、sudo 配置滥用、Cron 定时任务劫持、内核漏洞（Dirty Pipe） |
| **Windows** | 内核漏洞、服务权限配置错误、UAC 绕过、AlwaysInstallElevated、令牌窃取 |

#### 阶段五：横向移动与权限维持

拿下一台机器 ≠ 拿下一个网络。内网还有更多目标：

| 操作 | 目的 | 方法 |
|------|------|------|
| **横向移动** | 攻击同网段其他服务器 | SSH 密钥窃取、密码哈希抓取（mimikatz）、Pass-the-Hash、RDP 重放 |
| **权限维持** | 确保下次还能回来 | 创建隐藏账号、植入 SSH 后门、定时任务反弹 Shell、Webshell 留后门 |

### 永远在循环

这个五阶段流程不是线性的，而是一个**循环**：

- 横向移动后发现了新的内网服务器 → 回到阶段 1，重新扫描信息收集
- 提权后发现这个机器还有数据库 → 回到阶段 2，针对数据库找漏洞
- 拿下一个站后发现它是多站点服务器 → 回到阶段 3，对同服其他站点进一步利用

## 一、操作系统层面

### 1. 识别操作系统的常见方法

| 方法 | 具体操作 | 准确度 |
|------|----------|--------|
| **TTL 值判断** | `ping <目标IP>`，观察 TTL 返回值 | 粗略 |
| **Nmap -O** | `nmap -O <目标IP>`，TCP/IP 栈指纹匹配 | 高 |
| **抓包分析** | 分析 TCP 窗口大小、初始 TTL、IP ID 序列等 | 高 |
| **Banner 信息** | 连接 SSH、FTP 等服务，看返回的欢迎信息 | 中 |
| **HTTP 响应头** | 看 `Server` 字段暴露的信息 | 中 |
| **错误页面** | 访问不存在页面，触发 404/500 错误可能泄露 Server 信息 | 低 |

**TTL 值快速判断：**

| TTL 初始值 | 常见操作系统 |
|-----------|-------------|
| 64 左右 | Linux / Unix / macOS / Android |
| 128 左右 | Windows |
| 255 左右 | 网络设备（Cisco、路由器、防火墙） |

### 2. Windows 与 Linux 的核心区别及识别意义

| 维度 | Windows | Linux |
|------|---------|-------|
| **文件系统** | NTFS / FAT32 | ext4 / XFS / Btrfs |
| **路径分隔符** | `\`（反斜杠） | `/`（正斜杠） |
| **大小写敏感** | 不敏感（a = A） | 敏感（a ≠ A） |
| **常见 Web 服务** | IIS | Nginx / Apache |
| **常见数据库** | MSSQL | MySQL / PostgreSQL |
| **远程管理** | RDP（3389） | SSH（22） |
| **用户管理** | 域 / 本地用户 | UID/GID / root |
| **命令行** | cmd / PowerShell | bash / sh / zsh |
| **权限控制** | ACL / UAC | chmod / sudo |

**为什么要区分操作系统？**
- **漏洞方向不同**：Windows 关注 RDP、SMB、IIS 漏洞；Linux 关注 SSH、内核提权、Nginx/Apache 漏洞
- **利用方式不同**：反弹 Shell 的命令不同（`cmd.exe` vs `/bin/bash`）
- **提权路径不同**：Windows 走内核漏洞、服务权限、令牌窃取；Linux 走 SUID、sudo 配置、定时任务
- **后续横向移动方式不同**：Windows 域环境可用 Pass-the-Hash；Linux 通过 SSH 密钥传播

### 3. 操作系统层面常见漏洞类型

| 漏洞类型 | 典型代表 | 风险等级 |
|----------|----------|----------|
| **RCE 漏洞** | MS17-010（永恒之蓝）、CVE-2019-0708（蓝洞）、OpenSSL 心脏滴血 | 严重 |
| **权限提升** | Dirty Pipe（Linux）、PrintSpoofer（Windows）、内核提权 | 高 |
| **弱口令爆破** | RDP 弱口令、SSH 弱口令、FTP 弱口令 | 高 |
| **未授权访问** | SMB 共享未授权、Redis 无密码开放公网、各类服务默认口令 | 高 |
| **信息泄露** | NetBIOS 泄露主机名/用户名、SNMP 团体字泄露 | 中 |
| **补丁缺失** | 未打补丁且已公开利用方式的漏洞（Nday） | 严重 |
| **配置缺陷** | 不必要的服务端口暴露、危险端口直接对公网开放 | 中 |

**重点关注的 Windows 高危端口：**

| 端口 | 服务 | 攻击方向 |
|------|------|----------|
| 135 / 139 / 445 | RPC / NetBIOS / SMB | 远程代码执行（永恒之蓝）、密码爆破、共享枚举 |
| 3389 | RDP | 弱口令爆破、蓝洞 RCE、中间人攻击 |
| 1433 | MSSQL | 弱口令、XP_CMDSHELL 命令执行 |

**重点关注的 Linux 高危端口/服务：**

| 端口 | 服务 | 攻击方向 |
|------|------|----------|
| 22 | SSH | 弱口令爆破、密钥泄露 |
| 3306 | MySQL | 弱口令、SQL 注入综合利用 |
| 6379 | Redis | 未授权访问 → 写 SSH 公钥、写 Webshell、写定时任务 |
| 2375 | Docker API | 未授权访问 → 容器逃逸 |

### 4. 操作系统层面漏洞的影响范围

- **获取服务器完全控制权**：拿到最高权限（root / SYSTEM）
- **横向移动到内网**：以被控机器为跳板，攻击同网段的其他服务器
- **作为持久化据点**：植入后门、创建隐蔽账号、设置定时任务，长期保持访问
- **数据窃取与破坏**：拖库、加密勒索、删库跑路

> 操作系统一旦失陷，上面的所有应用（Web、数据库、缓存）全部沦陷。这是安全测试中收益最大、也是危害最高的环节。

## 二、数据库层面

### 1. 识别数据库类型的常见方法

| 方法 | 说明 |
|------|------|
| **端口扫描** | 每种数据库有默认监听端口，通过端口推断数据库类型 |
| **SQL 注入报错** | 不同数据库的报错信息有独特特征，可通过报错判断类型 |
| **Banner 信息** | 直接连接数据库服务获取欢迎信息 |
| **SQL 语法差异** | 利用不同数据库的独有函数/语法进行指纹识别 |
| **Web 应用特征** | CMS 类型可反推数据库（WordPress → MySQL） |

**常用数据库默认端口：**

| 数据库 | 默认端口 |
|--------|---------|
| MySQL | 3306 |
| MSSQL（SQL Server）| 1433 |
| Oracle | 1521 |
| PostgreSQL | 5432 |
| Redis | 6379 |
| MongoDB | 27017 / 27018 |
| Elasticsearch | 9200 |
| Memcached | 11211 |

**SQL 注入时通过函数指纹判断数据库类型：**

| 数据库 | 字符串拼接 | 注释符 | 特有函数/语法 |
|--------|-----------|--------|-------------|
| MySQL | `CONCAT()` 或 `||` | `#` `--` `/**/` | `sleep()` `load_file()` `into outfile` |
| MSSQL | `+` | `--` `/**/` | `xp_cmdshell` `waitfor delay` |
| Oracle | `\|\|` | `--` `/**/` | `utl_http.request()` `dbms_pipe.receive_message()` |
| PostgreSQL | `\|\|` | `--` `/**/` | `pg_sleep()` `COPY ... FROM PROGRAM` |
| Access | `&` | 无 | `CDate()` `IIF()` |

### 2. 数据库类型区别及识别意义

| 数据库 | 特点 | 常见搭配场景 |
|--------|------|-------------|
| **MySQL** | 开源、最流行的 Web 后端数据库 | PHP + Linux 站（LAMP/LNMP）|
| **MSSQL** | 微软出品，与 Windows/AD 深度集成 | ASP/ASPX/.NET + Windows 站 |
| **Oracle** | 大型企业级，功能强大但复杂 | Java 大型企业应用、金融系统 |
| **PostgreSQL** | 开源高级数据库，功能接近 Oracle | Python / Ruby / Go 应用 |
| **Redis** | 内存键值存储，常用于缓存/会话 | 各种 Web 应用的缓存层 |
| **MongoDB** | 文档型 NoSQL，JSON 格式存储 | Node.js / 大数据场景 |
| **Access** | 微软桌面数据库（.mdb 文件） | 小型 ASP 站点 |

**识别意义：**
- 不同数据库的**攻击方式**完全不同（注入语法、存储过程、利用工具都不同）
- 不同数据库的**提权路径**不同（MSSQL 可通过 xp_cmdshell 直接执行系统命令；MySQL 通常需要结合文件写入漏洞）
- 数据库类型暗示了**后端语言和操作系统**，帮助缩小攻击面

### 3. 数据库常见漏洞类型及攻击

| 漏洞类型 | 说明 | 典型案例 |
|----------|------|----------|
| **SQL 注入** | 未过滤用户输入，恶意拼接 SQL 语句 | 登录绕过、数据脱库、写入 Webshell |
| **弱口令/默认口令** | 数据库使用弱密码或出厂默认密码 | root/root、sa/sa、system/manager |
| **未授权访问** | 数据库对外开放且无需认证 | Redis 默认无密码、MongoDB 旧版默认无认证 |
| **CVE 漏洞** | 数据库软件本身的代码执行漏洞 | MySQL 任意文件读取、PostgreSQL 命令执行 |
| **存储过程滥用** | 利用数据库自带的危险存储过程 | MSSQL `xp_cmdshell` 执行系统命令 |
| **文件读写** | 利用数据库的文件操作函数 | MySQL `load_file()` 读文件、`INTO OUTFILE` 写 Webshell |
| **NoSQL 注入** | MongoDB/Redis 等非关系型数据库的注入 | `$ne` `$gt` 绕过、JSON 注入 |

**Redis 未授权访问三种利用方式：**

```
1. 写 SSH 公钥 → SSH 登录服务器
2. 写 Webshell → 通过 Web 访问执行命令
3. 写定时任务（crontab）→ 反弹 Shell
```

**MSSQL xp_cmdshell 执行命令：**
```sql
-- 启用 xp_cmdshell（如果被禁用）
EXEC sp_configure 'show advanced options', 1;
RECONFIGURE;
EXEC sp_configure 'xp_cmdshell', 1;
RECONFIGURE;

-- 执行系统命令
EXEC xp_cmdshell 'whoami';
```

### 4. 数据库层面漏洞的影响范围

- **数据泄露**：用户信息、密码哈希、交易记录被窃取
- **数据篡改/删除**：导致业务异常、数据丢失
- **权限提升**：从数据库权限提升到操作系统权限（通过存储过程或文件操作）
- **横向移动**：数据库服务器往往是内网核心节点，拿下后可进一步探测内网

> 数据库漏洞的"连锁反应"很强，一个 SQL 注入最终可能通向服务器完全控制。

## 三、第三方平台/软件层面

### 1. 如何判断目标使用了哪些第三方平台或软件

| 方法 | 说明 |
|------|------|
| **端口扫描** | 通过开放端口判断运行的服务（如 2181 → Zookeeper，9200 → Elasticsearch） |
| **HTTP 响应头/Cookie** | 如 `X-Powered-By: PHP/7.4`、`Set-Cookie: laravel_session=` |
| **页面特征/版权信息** | 页面底部"Powered by XXX"、favicon 哈希匹配 |
| **特定路径访问** | `/phpmyadmin/`、`/swagger-ui.html`、`/actuator/health` |
| **源码/静态文件分析** | JS 中引用的组件路径、CSS 注释中的框架信息 |
| **Wappalyzer / WhatWeb** | 使用浏览器插件或命令行工具自动识别指纹 |
| **错误信息泄露** | 应用报错可能暴露框架、组件、版本信息 |
| **目录扫描** | 扫描后台路径、管理页面、API 文档等 |

### 2. 为什么要识别第三方平台或软件

- **已知漏洞直接利用**：识别出 Struts2 → 直接找 S2 系列 RCE；识别出 ThinkPHP 5.x → 找历史 RCE
- **缩小测试范围**：不是每个站点都需要从头测 SQL 注入，如果是公开 CMS，直接查已知漏洞更高效
- **判断攻击深度**：知道用的是 Redis，可以关注未授权；知道用的是 Fastjson，可以关注反序列化
- **串联信息**：不同第三方软件的信息可以拼凑出完整的服务器架构图

### 3. 常见第三方平台/软件漏洞类型及攻击

| 软件类型 | 典型代表 | 常见漏洞 |
|----------|----------|----------|
| **Web 中间件** | Apache、Nginx、IIS、Tomcat、WebLogic | 解析漏洞、目录穿越、反序列化、弱口令 |
| **CMS** | WordPress、DedeCMS、帝国 CMS | SQL 注入、后台弱口令、任意文件上传、插件漏洞 |
| **开发框架** | Struts2、ThinkPHP、Spring、Fastjson | RCE、反序列化、表达式注入 |
| **数据库管理工具** | phpMyAdmin、Adminer | 弱口令、未授权访问、代码执行 |
| **缓存/消息队列** | Redis、RabbitMQ、ActiveMQ | 未授权访问、反序列化、弱口令 |
| **搜索服务** | Elasticsearch、Solr | 未授权访问、RCE、信息泄露 |
| **大数据组件** | Hadoop、Kafka、Zookeeper | 未授权访问、RCE |
| **CI/CD 工具** | Jenkins、GitLab | 未授权访问、RCE、凭证泄露 |
| **VPN/远程访问** | OpenVPN、AnyConnect、Fortinet VPN | CVE 漏洞、弱口令 |
| **虚拟化管理** | vSphere ESXi、Docker、K8s API | 未授权访问、容器逃逸、CVE 漏洞 |
| **邮件服务** | Exchange、Zimbra、Coremail | SSRF、RCE、信息泄露 |
| **负载均衡** | F5 BIG-IP、Nginx、HAProxy | CVE 漏洞、配置缺陷 |
| **OA/办公系统** | 泛微、致远、通达 OA | SQL 注入、RCE、任意文件上传 |

### 4. 第三方平台/软件安全测试的范围

不是只有 Web 应用才需要渗透测试，服务器上运行的所有服务都是潜在入口：

- **Web 相关**：Web 服务器、WAF、CDN、反向代理
- **数据库相关**：数据库本体、缓存、搜索引擎、管理工具
- **远程管理**：SSH、RDP、VNC、TeamViewer、堡垒机
- **邮件服务**：SMTP、POP3、Exchange、Webmail
- **文件服务**：FTP、SMB、NFS、Rsync
- **中间件**：消息队列、ESB、定时任务调度
- **监控运维**：Zabbix、Nagios、Grafana、Prometheus
- **容器编排**：Docker API、Kubernetes API
- **网络设备**：路由器、交换机、防火墙、负载均衡

**测试原则：** 每个暴露的服务都可能存在漏洞（弱口令、未授权访问、已知 CVE、配置缺陷），不放过任何一个开放端口。

## 四、补充：安全测试的全局视角

### 测试不仅仅是 Web

常规 Web 安全测试（SQL 注入、XSS、文件上传等）只是安全测试的一个子集。一个完整的渗透测试应该覆盖：

```
                    目标系统
                       │
        ┌──────────────┼──────────────┐
        │              │              │
    操作系统层       Web 应用层      第三方服务层
   （OS + 端口）    （源码 + CMS）   （中间件 + 组件）
        │              │              │
   · 永恒之蓝      · SQL 注入      · 反序列化
   · SSH 爆破      · XSS           · 未授权访问
   · 内核提权      · 文件上传      · CVE 漏洞
   · 弱口令        · 逻辑漏洞      · 弱口令
```

### 非典型 Web 服务的安全测试

有些目标不提供 Web 服务，仅提供单一或特定的网络服务，这些同样可以且应该被测试：

| 服务类型 | 示例 | 测试方向 |
|----------|------|----------|
| 邮件服务 | SMTP（25）、POP3（110）、IMAP（143） | 弱口令爆破、邮件伪造、SMTP 注入、Exchange CVE |
| 游戏服务 | 游戏服务器端口 | 协议逆向、封包篡改、逻辑漏洞 |
| 负载均衡 | F5 BIG-IP、Nginx | CVE 漏洞（如 F5 BIG-IP RCE）、配置缺陷 |
| 文件传输 | FTP（21）、Rsync（873） | 匿名登录、弱口令、目录遍历 |
| 数据库 | MySQL（3306）、MSSQL（1433） | 弱口令、未授权访问、存储过程利用 |
| 远程管理 | SSH（22）、RDP（3389）、VNC（5900） | 弱口令爆破、CVE 漏洞 |
| DNS | DNS（53） | DNS 域传送漏洞、DNS 隧道 |
| 打印服务 | IPP（631）、RAW（9100） | 打印机弱口令、打印任务窃取 |

**核心思路不变：** 不管目标是 Web 应用、API 接口、还是单纯的服务器服务，安全测试的底层逻辑都是一致的——**信息收集 → 版本识别 → 漏洞匹配 → 漏洞利用 → 权限维持**。只是具体的技术栈和工具不同而已。

## 五、CDN 绕过技术

### 什么是 CDN

CDN（Content Delivery Network，内容分发网络）将源站内容分发到全球各地边缘节点，用户访问时被解析到最近的 CDN 节点而非真实服务器。这对安全测试造成障碍——直接访问域名拿到的 IP 是 CDN 节点，不是真实服务器。

### 目前常见的 CDN 绕过技术

#### 1. 子域名查询

CDN 通常只绑定主域名（如 `www.example.com`），子域名往往直接解析到真实 IP。

| 方法 | 说明 |
|------|------|
| 爆破子域名 | 字典枚举 + DNS 解析，找到未使用 CDN 的子域 |
| 搜索引擎 | `site:*.example.com` 查找已索引的子域名 |
| 证书透明度日志 | crt.sh 查询 SSL 证书关联的所有域名 |
| 第三方平台 | SecurityTrails、VirusTotal、DNSdumpster 查询子域 |

```
# 常用工具
subfinder -d example.com
amass enum -d example.com
# 证书透明度查询
curl -s "https://crt.sh/?q=%.example.com&output=json" | jq .
```

#### 2. 邮件服务查询

很多网站使用自建邮件服务器或第三方邮件服务（如腾讯企业邮箱），发往该域名的邮件可能不会走 CDN。

| 方法 | 说明 |
|------|------|
| 注册/找回密码邮件 | 查看邮件源码中的 `Received` 头，追踪邮件路由路径 |
| MX 记录查询 | `dig example.com MX` 获取邮件服务器地址 |
| 邮件头分析 | 邮件头中 `X-Originating-IP` 等字段可能泄露真实 IP |
| 邮件订阅验证 | 部分网站的邮件验证链接指向真实服务器 |

```
# 查询 MX 记录
dig example.com MX
nslookup -type=MX example.com
```

#### 3. 国外地址请求

部分 CDN 服务商（尤其国内厂商）只覆盖中国大陆节点，使用国外 IP 访问时可能直接回源到真实服务器。

| 方法 | 工具/平台 |
|------|----------|
| 国外 VPS 直接访问 | 在境外服务器 `curl` 目标域名 |
| 在线全球 Ping 工具 | ping.pe、17ce.com、itdog.cn |
| 全球多地 HTTP 请求 | 使用全球节点发起 HTTP 请求对比响应差异 |
| DNS 解析对比 | 国内外不同 DNS 服务器解析结果对比 |

```
# 使用指定地区的 DNS 解析
nslookup example.com 8.8.8.8           # Google DNS（国外）
nslookup example.com 223.5.5.5         # 阿里 DNS（国内）
```

#### 4. 遗留文件与全网扫描

网站可能遗留了未经过 CDN 直连服务器的文件或服务。

| 方法 | 说明 |
|------|------|
| phpinfo 等探针文件 | 可能直接暴露 `SERVER_ADDR` 真实 IP |
| 网站备份文件 | 压缩包内的配置文件可能记录真实 IP |
| 全网扫描 | 通过特定 SSL 证书指纹、HTTP 响应特征全网匹配 |
| Favicon 哈希 | 计算网站 favicon 的哈希，通过 Shodan 搜索相同 favicon 的 IP |

```
# Shodan 搜索相同 SSL 证书
ssl:"<证书序列号>"
# Shodan 搜索相同 HTTP 标题
http.title:"<页面标题>"
```

#### 5. 黑暗引擎搜索特定文件

利用 Shodan、Fofa、ZoomEye 等网络空间搜索引擎，匹配网站的独有特征找到真实 IP。

| 引擎 | 搜索思路 |
|------|----------|
| **Shodan** | SSL 证书序列号、HTTP 响应体特征、页面关键词 |
| **Fofa** | `title="标题"` `body="特征字符串"` `cert="证书内容"` |
| **ZoomEye** | 与 Fofa 类似，网站特征指纹匹配 |
| **Censys** | SSL 证书链、HTTP 响应头特征 |

```
# Fofa 语法示例
title="某某后台管理系统" && country="CN"
body="<唯一特征字符串>"
cert="*.example.com"
```

#### 6. DNS 历史记录

CDN 通常是后加上去的，查询域名的 DNS 历史解析记录，可能找到绑定 CDN 之前的真实 IP。

| 方法 | 平台 |
|------|------|
| DNS 历史记录查询 | SecurityTrails、DNSdb、ViewDNS.info |
| 被动 DNS 数据 | VirusTotal、RiskIQ |
| DNS 缓存探测 | 向各地 DNS 服务器直接请求，查询是否缓存了旧记录 |

```
# 在线查询平台
# SecurityTrails: https://securitytrails.com/
# ViewDNS.info: https://viewdns.info/iphistory/
# DNSdb: https://dnsdb.io/
```

#### 其他绕过思路

| 方法 | 说明 |
|------|------|
| **CloudFlare 特定绕过** | 部分 CF 保护不当的站点，通过 `direct.example.com` 等 CNAME 暴露真实 IP |
| **SSRF 利用** | 网站存在 SSRF 漏洞时，让服务器自己请求自己，获取真实 IP |
| **敏感文件泄露** | `.git/config`、`.env`、`debug` 模式输出等泄露服务器内网 IP |
| **以量打量（DNS 泛洪）** | 对域名发起海量子域名请求，部分 CDN 因费用问题自动关闭防护回源 |

### CDN 真实 IP 获取后如何绑定访问

拿到真实 IP 后，需要让本地请求绕过 CDN 直接指向源站，否则访问域名还是走的 CDN。

**方法：更改本地 hosts 文件**

| 操作系统 | hosts 文件路径 |
|----------|---------------|
| Windows | `C:\Windows\System32\drivers\etc\hosts` |
| Linux / macOS | `/etc/hosts` |

```
# hosts 文件添加格式
<真实IP>  www.example.com
<真实IP>  example.com

# 示例
192.168.1.100  www.target.com
192.168.1.100  target.com
```

**生效方法：**
- Windows: `ipconfig /flushdns` 刷新 DNS 缓存
- Linux: `sudo systemctl restart nscd` 或 `sudo /etc/init.d/dns-clean`
- macOS: `sudo dscacheutil -flushcache; sudo killall -HUP mDNSResponder`

**注意事项：**
- 修改 hosts 后浏览器可能仍走 CDN 缓存，建议使用无痕模式或 `curl` 测试
- 测试完成后记得删除 hosts 记录，避免影响正常访问
- 部分 HTTPS 站点可能因证书不匹配报错（SNI 问题），可先确认 IP 后再进一步操作

## 六、课程体系与学习路径

### 漏洞技术课程安排

安全学习中漏洞技术是核心，按攻击面可以分为以下层次：

```
漏洞技术体系
├── Web 层面
│   ├── SQL 注入、XSS、CSRF、SSRF
│   ├── 文件上传、文件包含、命令注入
│   ├── 反序列化、模板注入
│   └── 逻辑漏洞（越权、支付、验证码绕过）
│
├── 系统层面
│   ├── 系统漏洞（永恒之蓝、内核提权）
│   ├── 端口服务漏洞（SMB、RDP、SSH）
│   └── 配置缺陷（弱口令、未授权访问）
│
├── 第三方 / 中间件层面
│   ├── Web 中间件（Apache、Nginx、IIS、Tomcat）
│   ├── 框架漏洞（Struts2、ThinkPHP、Fastjson）
│   ├── CMS 漏洞（WordPress、DedeCMS）
│   └── 组件漏洞（Redis、Elasticsearch、Docker）
│
└── 其他服务层面
    ├── 邮件服务（Exchange、SMTP）
    ├── 游戏服务（协议逆向）
    ├── 负载均衡（F5 BIG-IP）
    └── VPN / 远程接入（Fortinet、Citrix）
```

### 安全测试的核心思路

不管目标是什么类型，安全测试的底层逻辑保持一致：

1. **信息收集** — 端口扫描、服务识别、指纹匹配
2. **版本识别** — 确定操作系统、中间件、数据库、框架的具体版本
3. **漏洞匹配** — 根据版本查找公开 CVE / Nday / 配置缺陷
4. **漏洞利用** — 使用 PoC/Exp 进行验证和利用
5. **权限维持** — 建立持久化访问通道

> 只要理清了这个思路，无论是 Web 应用、系统服务、还是其他复杂服务（邮件、游戏、负载均衡等），安全测试的方法论都是通用的。

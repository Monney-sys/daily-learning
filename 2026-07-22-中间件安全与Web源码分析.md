# 2026-07-22 中间件安全与 Web 源码分析

## 一、中间件安全

### 什么是中间件

中间件（Middleware）是位于操作系统和应用程序之间的服务器软件，负责处理 HTTP 请求、解析脚本、管理会话等核心功能。常见的中间件包括：

| 中间件 | 开发商 | 常见运行环境 | 脚本语言 |
|--------|--------|-------------|----------|
| **Apache HTTP Server** | Apache 基金会 | Linux / Windows | PHP（最常搭配） |
| **Nginx** | Nginx Inc. | Linux | PHP、Python、Go（反向代理） |
| **IIS**（Internet Information Services）| 微软 | Windows Server | ASP / ASPX / .NET |
| **Tomcat** | Apache 基金会 | Linux / Windows | Java / JSP |
| **WebLogic** | Oracle | Linux / Windows | Java |
| **JBoss / WildFly** | Red Hat | Linux / Windows | Java |

### 中间件常见安全漏洞

#### 1. Apache

| 漏洞类型 | 说明 | 风险等级 |
|----------|------|----------|
| **多后缀解析漏洞** | `test.php.xxx` 被解析为 PHP 执行 | 高 |
| **.htaccess 文件利用** | 通过上传恶意 `.htaccess` 修改解析规则 | 高 |
| **目录穿越（CVE-2021-41773）** | 路径规范化绕过，读取任意文件 | 严重 |
| **SSI 注入** | 服务端包含注入，可执行系统命令 | 中 |
| **CORS 配置不当** | 允许任意域跨域访问 | 中 |

#### 2. Nginx

| 漏洞类型 | 说明 | 风险等级 |
|----------|------|----------|
| **CRLF 注入** | 在响应头中注入恶意内容、XSS | 中 |
| **路径解析不当** | `/1.jpg/1.php` 被当作 PHP 执行（旧版本） | 高 |
| **目录穿越** | `alias` 配置不当导致目录穿越 | 高 |
| **HTTP 走私** | 请求边界歧义，绕过安全设备 | 高 |

#### 3. IIS

| 漏洞类型 | 说明 | 风险等级 |
|----------|------|----------|
| **PUT 任意文件上传** | WebDAV + PUT 方法配置不当，可上传任意文件 | 高 |
| **短文件名猜解** | Windows 8.3 短文件名机制可被暴力枚举 | 中 |
| **解析漏洞** | `test.asp;.jpg` 被解析为 ASP；`test.asp/` 目录下任意文件被解析为 ASP | 高 |
| **IIS 6.0 文件名解析** | `*.asp;.jpg` 被当作 ASP 执行 | 高 |

#### 4. Tomcat

| 漏洞类型 | 说明 | 风险等级 |
|----------|------|----------|
| **弱口令 + War 包部署** | manager 页面弱口令导致远程部署 webshell | 严重 |
| **CVE-2017-12615** | PUT 方法任意文件上传 | 高 |
| **AJP 协议漏洞（Ghostcat）** | AJP 协议未授权访问，读取源码或 RCE | 严重 |
| **样例文件信息泄露** | 默认安装的 Examples 应用暴露服务器信息 | 低 |

#### 5. WebLogic

| 漏洞类型 | 说明 | 风险等级 |
|----------|------|----------|
| **反序列化漏洞** | T3/IIOP 协议反序列化导致远程命令执行 | 严重 |
| **SSRF** | 服务器端请求伪造，扫描内网服务 | 高 |
| **控制台弱口令** | 默认口令或弱口令登录，部署 War 包 | 严重 |
| **CVE-2020-14882** | 未授权访问后台，进一步 RCE | 严重 |

> **核心思路：** 渗透测试中，首先要识别目标使用的是哪种中间件及其版本号，然后查找该版本是否存在已知漏洞（CVE）。不同中间件的漏洞利用方式差异很大，但"信息收集 → 版本识别 → 漏洞匹配"这个流程是通用的。

## 二、Vulhub — 开源漏洞复现靶场

### 简介

[Vulhub](https://github.com/vulhub/vulhub) 是一个面向大众的开源漏洞环境集合，由 phith0n 创建维护。它基于 Docker 和 Docker Compose，为大量已知漏洞提供了**一键搭建的复现环境**，是目前最流行的漏洞学习平台之一。

### 特点

- **开箱即用**：只需 `docker-compose up -d` 即可启动漏洞环境
- **覆盖广泛**：涵盖 WebLogic、Tomcat、Apache、Nginx、Struts2、Redis、MySQL 等数百个漏洞
- **环境隔离**：基于 Docker，不影响宿主机，用完即删
- **学习友好**：每个漏洞目录下有对应说明文档

### 涵盖的漏洞类型

| 分类 | 示例 |
|------|------|
| Web 应用漏洞 | SQL 注入、XSS、SSRF、文件上传、RCE |
| 中间件漏洞 | Apache、Nginx、Tomcat、WebLogic、IIS |
| 框架漏洞 | Struts2、ThinkPHP、Spring、Django |
| CMS 漏洞 | WordPress、Drupal、DedeCMS |
| 服务漏洞 | Redis、MySQL、PostgreSQL、Docker |
| 协议漏洞 | HTTP 走私、AJP（Ghostcat）、Fastjson |

### 实际操作体验

今天尝试复现了 **Apache 多后缀名解析漏洞**，核心原理：

```
Apache 在处理不认识的文件扩展名时，会从右向左逐级回退查找可识别的解析规则。

文件: test.php.xxx
步骤1: 找不到 .xxx 的解析器
步骤2: 回退到 .php → 匹配到 PHP 解析器
步骤3: 文件被当作 PHP 脚本执行！

攻击者上传一个名为 shell.php.xxx 的文件，
即可绕过只检查 ".php" 后缀的上传限制，最终获得 webshell。
```

**防御方式**：
- 使用白名单限制允许上传的后缀名
- Apache 配置中禁用 `AddHandler` 的通配回退行为
- 重命名上传文件，不给文件保留原始扩展名

## 三、Web 源码安全分析

渗透测试中拿到一个目标网站后，分析其源码结构是重要的信息收集环节，可以从以下维度切入：

### 1. 脚本语言识别

不同脚本语言有各自的文件扩展名和漏洞特征：

| 语言 | 文件后缀 | 常见漏洞类型 |
|------|----------|-------------|
| **PHP** | `.php` `.phtml` `.pht` `.php3` `.php5` | 文件包含（LFI/RFI）、SQL 注入、任意文件上传、代码执行、反序列化 |
| **ASP** | `.asp` | SQL 注入、文件上传、Access 数据库下载 |
| **ASPX / .NET** | `.aspx` `.ashx` `.asmx` `.ascx` | SQL 注入、反序列化、ViewState 篡改、Web.config 泄露 |
| **JSP** | `.jsp` `.jspx` `.do` `.action` | Struts2 RCE、反序列化、表达式注入（EL 注入）、未授权访问 |
| **Python** | 无固定后缀（路由驱动） | SSTI（模板注入）、反序列化、命令注入、Flask/Django 调试页泄露 |

**识别方法：**
- 查看 URL 后缀名（如 `.php`、`.aspx`、`.jsp`）
- 查看 HTTP 响应头 `Server` 字段和 `X-Powered-By` 字段
- 访问不存在的页面触发报错（可能暴露框架信息）

### 2. 网站类型判断

| 网站类型 | 特征 | 关注点 |
|----------|------|--------|
| **门户网站** | 功能全面，模块多，用户量大 | 后台地址、编辑器漏洞、评论 XSS |
| **企业网站** | 展示为主，功能较少 | 后台弱口令、信息泄露、旁站攻击 |
| **电商网站** | 购物车、支付、会员系统 | 支付逻辑漏洞、SQL 注入、越权漏洞 |
| **论坛/BBS** | 发帖、评论、私信 | XSS、SQL 注入、CSRF、越权 |
| **博客** | 文章发布、评论 | CMS 版本漏洞、评论 XSS、弱口令 |

### 3. 框架 vs 非框架

#### 使用框架开发

| 框架 | 语言 | 常见漏洞 |
|------|------|----------|
| ThinkPHP | PHP | 远程代码执行（历史版本频发）、SQL 注入、缓存文件泄露 |
| Laravel | PHP | 反序列化、`.env` 配置泄露、Debug 模式信息泄露 |
| Spring / Spring Boot | Java | Actuator 未授权、反序列化、SSTI、SpEL 注入 |
| Struts2 | Java | S2 系列 RCE 漏洞（历史重灾区） |
| Django | Python | SQL 注入、Debug 模式泄露、CSRF 配置不当 |
| Flask | Python | SSTI（Jinja2 模板注入）、Debug 模式 RCE |

**识别框架的方法：**
- URL 路由风格（如 ThinkPHP 的 `/index.php/Home/Index/index`）
- 错误页面特征（Laravel 的 Whoops 页面、Spring Boot 的 Whitelabel Error Page）
- Cookie 名称（如 `laravel_session`、`JSESSIONID`、`PHPSESSID`）
- 响应头中的 `X-Powered-By`、`Server`
- 网站根目录下的 `robots.txt`、`composer.json`、`package.json` 等文件

#### 非框架（原生手写）

- **优点（从攻击者角度）：** 通常缺乏安全防护，容易出现 SQL 注入、XSS 等基础漏洞；没有固定的漏洞模式，不需要关注框架 Nday
- **缺点：** 代码结构不统一，需要手动分析，自动化程度低
- **攻击思路：**
  - 重点测试基础漏洞：SQL 注入、XSS、文件上传、命令执行
  - 关注管理后台：弱口令、未授权访问
  - 测试逻辑漏洞：越权、支付漏洞、验证码绕过

### 4. 网站目录结构分析

拿到网站源码或通过目录扫描后，需要能看懂每个文件和目录的作用，这对定位漏洞至关重要。以下是常见项目中各类文件/目录的含义：

---

#### 4.1 通用敏感文件

这些文件几乎在所有类型的项目中都可能出现：

| 文件 | 含义 | 渗透价值 |
|------|------|----------|
| `robots.txt` | 爬虫协议文件，声明哪些路径不允许搜索引擎爬取 | **高** — 往往直接暴露后台地址、敏感目录 |
| `sitemap.xml` | 网站地图 | 中 — 可以了解网站整体结构 |
| `.env` | 环境变量配置文件 | **极高** — 可能包含数据库密码、API 密钥、加密盐值等 |
| `.git/config` | Git 仓库配置 | **高** — 包含远程仓库 URL，仓库地址可能含凭证 |
| `composer.json` | PHP 依赖声明 | 中 — 暴露依赖库及版本，可查已知漏洞 |
| `composer.lock` | PHP 依赖锁定文件 | 高 — 记录每个依赖的精确版本号 |
| `package.json` | Node.js 依赖声明 | 中 — 暴露依赖和自定义脚本 |
| `package-lock.json` | Node.js 依赖锁定文件 | 高 — 记录精确版本 |
| `requirements.txt` | Python 依赖列表 | 中 — 暴露 Python 包和版本 |
| `Gemfile` / `Gemfile.lock` | Ruby 依赖 | 中 — 暴露 Ruby 包版本 |
| `Dockerfile` | Docker 镜像构建文件 | 中 — 了解运行环境配置 |
| `docker-compose.yml` | Docker 多容器编排 | **高** — 可能包含数据库密码等敏感信息 |
| `.DS_Store` | macOS 系统文件 | 低 — 可能泄露目录结构 |
| `README.md` / `CHANGELOG.md` | 项目说明 / 变更日志 | 低 — 了解项目背景和版本 |
| `LICENSE` | 开源许可证文件 | 低 |

---

#### 4.2 `.git` 源码泄露

如果服务器的 `.git` 目录被暴露（常见于部署不规范导致），攻击者可以用工具（GitHack、dvcs-ripper 等）下载整站源码。这是 **信息收集阶段的重大发现**，可以直接获取：

- 所有源代码文件
- 数据库连接配置（password、host、dbname）
- API 密钥、加密盐值
- 管理员账号密码的哈希值
- 代码中的注释和调试信息

#### 4.3 PHP 项目常见结构

```
项目根目录/
├── index.php              # 网站入口文件，所有请求通常经过这里
├── config.php             # 核心配置文件（数据库账号密码！）
├── config.inc.php         # 配置文件（旧项目常见写法）
├── db.php / database.php  # 数据库连接文件，包含连接凭据
├── install/               # 安装目录 — 安装完成后未删除可重装整站！[高危]
├── admin/                 # 后台管理目录
├── api/                   # API 接口目录
├── uploads/ / upload/     # ⚠ 用户上传文件存放目录
├── include/ / inc/        # 公共函数/类库目录（通常无法直接访问）
├── template/ / templates/ # 模板文件目录
├── theme/ / themes/       # 主题/皮肤目录
├── cache/ / temp/ / tmp/  # 缓存和临时文件目录
├── data/                  # 数据目录（DedeCMS 等国产 CMS 常见）
├── logs/                  # ⚠ 日志目录，可能记录请求、错误信息
├── backup/ / bak/ / old/  # ⚠ 备份目录，可能包含源码备份包
├── static/ / assets/      # 静态资源（CSS/JS/图片）
├── .htaccess              # Apache 目录级配置（可被利用）
├── phpinfo.php            # ⚠ PHP 环境信息输出测试文件
├── test.php / info.php    # ⚠ 测试文件，可能暴露敏感信息
└── vendor/                # Composer 第三方依赖目录
```

| | `wp-content/uploads/` | 上传文件目录 |

| | `xmlrpc.php` | XML-RPC 接口，可被用于爆破 |
| **DedeCMS** | `data/` | 数据目录，可能存有数据库备份 |
| | `dede/` | 后台目录 |
| | `plus/` | 扩展功能目录 |
| | `uploads/` | 上传目录 |

#### 4.4 Java 项目常见结构

```
项目根目录/
├── WEB-INF/
│   ├── web.xml             # ⚠ 应用部署描述文件，映射所有 URL 路由和过滤器
│   ├── classes/            # 编译后的 Java 类文件

│   └── applicationContext.xml  # Spring 配置文件

├── application.properties  # Spring Boot 配置（可能含数据库密码）
├── application.yml         # Spring Boot 配置（可能含数据库密码）
└── actuator/               # ⚠ Spring Boot Actuator 端点，可能泄露配置信息
```

| 文件/目录 | 说明 | 风险 |
|-----------|------|------|

> **总结：** 看懂目录结构是渗透测试的基本功。每个文件都可能泄露一条有用信息——配置文件中的密码、备份包中的源码、日志中的请求记录。实际渗透中要把这些零散信息串联起来，逐步扩大战果。

## 四、CMS（内容管理系统）

### 概念

CMS（Content Management System，内容管理系统）是一类用于快速建站的**成品网站系统**。使用者无需编写代码，通过后台管理界面即可发布文章、管理页面、修改模板，常见于企业站、博客、资讯站等。

### 主流 CMS

| CMS | 语言 | 特点 |
|-----|------|------|
| **WordPress** | PHP | 全球市场占有率最高（约 43%），插件生态丰富 |
| **DedeCMS（织梦）** | PHP | 国内老牌 CMS，历史漏洞非常多 |
| **帝国 CMS** | PHP | 国内使用广泛，安全口碑相对较好 |
| **Discuz** | PHP | 论坛 CMS，曾经被广泛使用 |
| **Drupal** | PHP | 国外企业级 CMS，美国政府常用 |
| **Joomla** | PHP | 国外开源 CMS |
| **PHPCMS** | PHP | 国内曾经的流行 CMS |

### CMS 对渗透测试的意义

- **提高效率**：识别出 CMS 后，可以直接查找该 CMS 的已知漏洞（Nday），而不需要手动逐个测试
- **漏洞集合集中**：目标使用同一套 CMS 时，一个漏洞可以通杀所有站点
- **识别方式**：
  - 网站底部版权信息（如 "Powered by DedeCMS"）
  - 后台路径（如 `/wp-admin/`、`/dede/`）
  - 页面源码中的特征字符串
  - 第三方 CMS 指纹识别工具（如 WhatWeb、Wappalyzer 插件等）

# 2026-07-21 数据包与 Burp Suite 抓包实战

## 一、数据包结构

### HTTP 数据包的完整组成

```
┌─────────────────────────────────┐
│         请求行 / 状态行           │
├─────────────────────────────────┤
│         请求头 / 响应头           │
│         (Headers)               │
├─────────────────────────────────┤
│          空行 (CRLF)            │
├─────────────────────────────────┤
│         消息体 (Body)            │
│       (POST/PUT 请求的数据)      │
└─────────────────────────────────┘
```

### 请求行结构
```
方法  URL  协议版本
GET   /index.php?id=1   HTTP/1.1
```

### 请求方法（重点）

| 方法 | 用途 | 特点 |
|------|------|------|
| **GET** | 获取资源 | 参数拼接在 URL 中，明文可见，有长度限制 |
| **POST** | 提交数据 | 参数放在 Body 中，相对隐蔽，无长度限制 |
| **PUT** | 更新资源（全量） | 替换整个资源 |
| **DELETE** | 删除资源 | 删除指定资源 |
| **PATCH** | 更新资源（部分） | 只更新指定字段 |
| **HEAD** | 获取响应头 | 不返回 Body，用于检查资源是否存在 |
| **OPTIONS** | 查询支持的方法 | 跨域预检请求常用 |

### GET 与 POST 的核心区别

| | GET | POST |
|------|------|------|
| 参数位置 | URL 查询字符串 | 请求体 Body |
| 安全性 | 低（URL 明文） | 相对高 |
| 数据量 | 受 URL 长度限制（约 2KB） | 无限制 |
| 缓存 | 可被缓存 | 不会被缓存 |
| 回退 | 无副作用，可重复 | 可能产生副作用 |

## 二、常见请求头参数

| 请求头 | 说明 | 示例 |
|--------|------|------|
| `Host` | 目标主机域名 | `www.example.com` |
| `User-Agent` | 客户端浏览器/系统信息 | `Mozilla/5.0 ...` |
| `Cookie` | 会话凭证 | `sessionid=abc123` |
| `Referer` | 请求来源页 URL | `https://www.baidu.com/` |
| `X-Forwarded-For` | 客户端真实 IP（代理场景） | `192.168.1.1` |
| `X-Real-IP` | 真实 IP（Nginx 常用） | `192.168.1.1` |
| `Content-Type` | 请求体数据类型 | `application/json` |
| `Authorization` | 认证令牌 | `Bearer tokenxxx` |
| `Origin` | 跨域请求来源 | `https://a.com` |
| `Accept` | 客户端接受的响应类型 | `text/html,application/json` |

## 三、Burp Suite 抓包入门

### Burp Suite 简介
Burp Suite 是 Web 安全测试的核心工具，由 PortSwigger 开发。主要功能：
- **Proxy**：拦截和修改 HTTP/HTTPS 请求
- **Repeater**：重复发送和修改请求
- **Intruder**：批量自动化测试（爆破、遍历）
- **Decoder**：编解码工具

### 基本使用流程
1. 配置浏览器代理为 `127.0.0.1:8080`
2. 开启 Burp Suite 拦截（Intercept is on）
3. 浏览器访问目标网站，Burp 捕获请求
4. 右键 → Send to Repeater 进行手动修改重放
5. 修改请求参数后点击 Send 查看响应

## 四、靶场实战

### 实战一：伪造来源页（Referer 绕过）

**场景**：某网站后台只允许从 `admin.example.com` 来源访问。

**原理**：服务端通过检查 `Referer` 请求头判断请求来源，但该字段由客户端发送，可被伪造。

**操作步骤**：
1. Burp Suite 拦截请求
2. 将 `Referer: https://normal.example.com/` 修改为 `Referer: https://admin.example.com/`
3. Forward 发送请求，成功绕过来源验证

**防御建议**：不要依赖 Referer 做安全校验，应使用 Token 或 Session 等不可伪造的机制。

### 实战二：随机 IP 进行投票刷票

**场景**：某投票系统通过 IP 地址限制每人只能投一票。

**原理**：服务器通常通过 `X-Forwarded-For` 或 `X-Real-IP` 头获取客户端 IP，这些头可以被客户端伪造。

**操作步骤**：
1. Burp Suite 拦截投票请求
2. 添加请求头：`X-Forwarded-For: <随机IP>`
3. 使用 Intruder 模块，将 IP 部分设为变量
4. 加载 IP 字典或使用随机数生成
5. 启动攻击，每次发送不同 IP 绕过限制

**防御建议**：
- 不要信任客户端传来的 IP 头
- 从 TCP 连接层获取真实 IP
- 结合账号体系限流（每个账号只能投一次）

### 实战三：修改请求参数绕过前端校验

**场景**：前端对投票数量做了限制，但后端未做校验。

**操作步骤**：
1. 抓包获取投票请求
2. 修改请求体中的投票数量参数
3. 绕过前端的 JS 限制，直接发送修改后的数据

**防御建议**：前端校验只是为了用户体验，真正的安全校验必须在后端实现。

## 五、总结

> 核心认知：**所有客户端传来的数据都不可信。**
>
> HTTP 请求头中的 `Referer`、`X-Forwarded-For`、`User-Agent`、`Cookie` 等所有字段都可以被任意修改。安全校验绝不能仅依赖这些字段，必须在服务端建立可靠的验证机制。
